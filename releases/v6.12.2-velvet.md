# v6.12.2-velvet — Server Speaker & Reliability Fixes

**Release date:** April 21, 2026

> A focused bugfix release targeting the Server Audio (MPV cast-to-speaker) feature,
> the automatic library scan scheduler, and the scan-progress indicator on the home screen.

---

## Bug fixes

### fix: Cast-to-speaker button stays white after page refresh
- `S.castingToMpv` was an in-memory flag only — a browser refresh always reset it to `false`,
  leaving the cast button white even though MPV was still playing.
- Fix: cast state is now persisted to `localStorage` (`ms2_casting_mpv_<user>`).
  On boot `_refreshServerAudioState()` checks the flag, confirms MPV is still playing via
  `/api/v1/server-playback/status`, and re-activates cast mode (mutes browser audio,
  turns button green) without any user action.

### fix: Cast-to-speaker not restored after server restart
- After a server restart MPV's queue is empty. The previous code saw `playing: false`
  and cleared the cast flag, treating it as if the user had stopped casting.
- Fix: `_refreshServerAudioState()` now distinguishes three cases:
  1. **MPV playing** (browser refresh) — re-mute browser, restore green button.
  2. **MPV running but empty** (server restart) — re-load the current queue song into MPV
     at the last saved position, re-mute browser, restore green button.
  3. **MPV not running** — clear the stale flag.

### fix: Cast-to-speaker did not disable browser crossfade
- Enabling cast while browser crossfade was active caused both the browser and MPV to
  attempt fades simultaneously, producing double-fade artefacts and cut-off tracks.
- Fix: activating cast now saves the current crossfade value and sets it to 0 (same
  mechanism as gapless mode). Deactivating cast restores the saved value.

### fix: MPV crossfade caused songs to be cut off
- The Node.js-side crossfade implementation polled `time-pos` over IPC and ramped volume
  in 50 ms steps. Because MPV advances its playlist on its own clock, the song ended
  before the fade completed, cutting audio abruptly.
- Fix: crossfade code removed entirely from the server-side MPV path.
  `--gapless-audio=yes` is kept (removes the inter-track silent gap cleanly).

### fix: MPV keeps playing when browser tab is closed
- Closing or navigating away from `/server-remote` left MPV playing indefinitely because
  no pause command was sent.
- Fix: `pagehide` and `beforeunload` handlers in the server-remote SPA now fire a
  synchronous XHR `POST /api/v1/server-playback/set-pause` before the page unloads.

### fix: Scheduled library scan never ran after a server restart
- `setInterval` resets its clock on every process start. With `bootScanEnabled: false`,
  no scan fires at boot, and restarts closer than 12 h apart meant the 12-hour window
  was continuously reset — scans could be skipped for days.
- Fix: `runAfterBoot()` in `task-queue.js` now compares `Date.now()` against the last
  recorded scan timestamp. If the gap exceeds the configured interval the scan fires
  immediately on boot, regardless of `bootScanEnabled`.

### fix: Scan progress bar did not appear automatically after boot scan started
- `pollScan()` used a one-shot `setTimeout` chain that stopped polling when no scan was
  active. A scan that started after the initial poll (e.g. the overdue boot scan above)
  was never detected — users had to manually refresh the page to see progress.
- Fix: `pollScan()` now always reschedules itself — every 3 s while a scan is active
  (unchanged), every 15 s while idle. The progress bar appears automatically within
  15 seconds of any scan starting, with no page refresh.
