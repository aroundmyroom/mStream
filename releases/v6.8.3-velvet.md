# v6.8.3-velvet — April 2026 — Playback stability + artist index performance

`v6.8.3` is a bugfix release focused on one core issue: intermittent playback stalls caused by heavy artist-index rebuild work running on the main server thread.

---

## Fix: artist rebuild no longer blocks audio streaming

`rebuildArtistIndex()` now runs its heavy normalization and rebuild work in a dedicated `worker_thread`.

Before this change, large libraries could spend many seconds in CPU-heavy artist grouping on the main event loop. During that window, stream responses could be delayed long enough for browser buffers to run dry, causing:

- `Stream stalled` messages in the browser console
- repeated recovery attempts
- occasional audible stop/resume during playback

Result in `v6.8.3`:

- audio-serving event loop stays responsive while rebuild runs
- playback continues normally during artist index maintenance
- duplicate concurrent rebuild triggers are deduplicated while one rebuild is already running

---

## Fix: instant Admin Artists On/Off feedback

In **Admin → Directories**, toggling **Artists: On/Off** now responds immediately:

- UI updates optimistically right away
- API returns immediately after config save
- rebuild happens in the background
- artist cache invalidates after rebuild completion

This removes the long delay that previously encouraged repeated button clicks on large libraries.

---

## Performance: new SQL indexes for artist hot paths

Added and verified:

- `idx_an_song_count` on `artists_normalized(song_count DESC)`
- `idx_an_image_flag` on `artists_normalized(image_flag_wrong, song_count DESC)`
- `idx_files_vpath_artist` on `files(vpath, artist)`

These improve:

- Artist Home top-list sorting
- Admin image-audit list retrieval
- artist-song browse queries scoped by vpath

All changes are backward-compatible and applied automatically on startup via `CREATE INDEX IF NOT EXISTS`.
