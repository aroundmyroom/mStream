# v6.11.2-velvet — Performance & Bug-Fix Bundle

**Release date:** April 19, 2026

Bug-fix and performance release. No new major features; focuses on correctness issues found after v6.11.0/6.11.1 and several page-load performance regressions that became apparent with larger queues and active play histories.

---

## New features

### Queue: remaining time + per-track estimated start times

- The **Up Next** counter now shows total remaining playback time when Auto-DJ is off — e.g. `(12 · 47:32)`. With Auto-DJ on the count stays plain (the queue shifts constantly so a total would be misleading).
- Hovering any upcoming queue row shows an estimated start time tooltip (`~14:47`) — cumulative from the current playback position, updated on each `refreshQueueUI()`.

### Context menu: Go to Artist / Go to Album

Two new items in the right-click/long-press song context menu:
- **Go to Artist** — navigates to the artist's album list in the Album Library.
- **Go to Album** — opens the album's song list.

Hidden automatically for radio stations, podcasts, and tracks with no artist/album tag.

### Playlist total duration

Playlist views now show a metadata line above the song list: **42 songs · 2:47:13**.

### Smart playlist sidebar tooltip

Each smart playlist button in the sidebar now has a tooltip summarising its active filters — genres, year range, minimum rating, played/unplayed, play-count threshold, starred, artist filter, sort order, and item limit.

## Bug fixes

### Album songs missing duration in queue

Songs added from the Album Library had `duration: null` in the queue — the internal `_albSongObj()` builder was not copying the `duration` field from the track row. This caused three visible problems: no duration shown on queue rows, no estimated-start tooltip, and the Up Next remaining-time total was incorrect.

### Album library sort — series cards now respect sort order

The sort selection applied only to standalone album cards; album series cards were always rendered A–Z regardless. Fixed by unifying series and standalone albums into a single sortable list. The sort `<select>` dropdown has also been replaced by a compact cycle button: **A–Z → Z–A → ↓ Year → ↑ Year**.

### WAV / AIFF: audio format info missing from queue panel

Two separate bugs prevented WAV and AIFF tracks added from the Album Library from showing their bitrate/kHz/channel info in the queue now-playing card:

1. The async metadata fetch was gated on `s['sample-rate'] == null`, but album-browse rows arrived with the field absent (not `null`). The trigger condition now correctly handles both `null` and `undefined`.
2. The `/api/v1/db/metadata` response wraps data inside a `metadata` key; the read path was accessing the top-level object directly. Fixed with `meta?.metadata || meta` fallback.

## Performance fixes

### Art fetch storm with large queues (100+ songs)

When `restoreQueue()` ran (called twice on every boot — once from `_applyServerSettings`, once from the boot sequence), each call triggered `refreshQueueUI()` → `_fetchMissingArt()`. For every song without cached album art, a GET request to `/api/v1/files/art` was fired. With 100 queued songs this produced **200 concurrent API calls**, exhausting the browser's HTTP connection pool and stalling the entire page load.

Fixed by adding a `_artFetchInFlight` Set: if a fetch for a given filepath is already in-progress, subsequent calls skip it silently. Art arrives once, patches every matching row via DOM query.

### refreshQueueUI — single O(n) pass (was 2×)

The Up Next remaining time was computed with `Array.slice + reduce` (allocating a sub-array) and the per-track start offsets in a separate `for` loop — two separate O(n) passes per call. `refreshQueueUI()` is called from 20+ places. Merged into one loop with no array allocation: both `_remainSecs` and `_startOffsets` are computed together.

### Home view — listening streak no longer blocks the event loop

`getHomeSummary()` computed the listening streak by firing up to **365 separate SQLite queries** — one per calendar day — in a synchronous loop on the Node.js event loop. For a user with a 100-day streak, this meant 100 sequential DB calls before the endpoint could respond.

Replaced with a **single** `SELECT DISTINCT CAST(started_at / 86400000 AS INTEGER) … ORDER BY b DESC LIMIT 366` query; streak counting runs in JS from the returned buckets. Always 1 DB query regardless of streak length.

### Home view — recently-added fetch limit reduced

The Home view fetched up to **300 recently-added songs** (`_limit × 20`) to build `_limit` album groups (~12 visible cards). Only the first `_limit` groups are ever displayed — the rest were discarded. Limit reduced to `_limit × 6` (max 90 songs): same display result, ~70% less data transferred and parsed on every home-page load.
