# v6.13.3-velvet — April 2026 — Custom Placeholder, Similar Artists & Queue Performance

## Overview

v6.13.3 ships five improvements across artist image management, the admin panel, and playback performance: a customisable artist placeholder image (upload your own via Admin), a dynamic public API endpoint for it, deduplication of similar-artist chips that include collaboration suffixes, a fix for feat./ft. artist tags breaking library lookups, and a significant queue-add performance fix that eliminates background ffmpeg jobs when adding large numbers of songs.

---

## New: Customisable artist placeholder image

When an artist has no photo yet, mStream Velvet shows a placeholder image. You can now **replace the built-in placeholder with your own image** from the Admin panel.

### How it works

- Open **Admin → Artist Images** — a new card at the top shows the current placeholder as a circular preview
- Click **Upload image** to pick any image file; mStream Velvet resizes it to 400×400 JPEG and stores it in `image-cache/artist-placeholder.jpg`
- The placeholder is served via a new public endpoint (`GET /api/v1/artists/placeholder`) that all four artist image surfaces use — Artist Home, Artist Library, Artist Profile, and Playing Now sidebar
- Click **Delete custom** to remove your image and revert to the built-in default

> **After uploading or deleting:** force-refresh your browser (Ctrl+Shift+R / Cmd+Shift+R) to clear the cached image.

---

## New: Artist placeholder served via public API

All artist fallback images in the player now use `api/v1/artists/placeholder` instead of the static `assets/img/unknownartist.webp`. This means:

- Uploading a custom placeholder updates it everywhere — no code change, no restart
- The endpoint requires no authentication (images must be loadable in `<img src>` tags)
- Falls back to the built-in webp automatically if no custom image has been uploaded

## Fix: feat./ft. artist tags no longer break library lookups

When the currently playing song has a collaboration tag like `"Calvin Harris feat. Ellie Goulding"`, the Playing Now panel was passing the full string to every lookup — resulting in "In Your Library" showing 1 song instead of Calvin Harris's full catalogue, and the wrong (or no) artist image.

A new `_primaryArtist()` helper strips `feat.`, `ft.`, `featuring`, `vs.`, `with`, `pres.`, and `×` suffixes before passing the name to artist image, library, Last.fm bio, and similar-artist lookups. Display names are unchanged.

---

## Fix: Similar artist chips — deduplication of collaboration variants

The similar-artists list from Last.fm sometimes returns both `"Blank & Jones"` and `"Blank & Jones ft. Robert Smith"` as separate entries. These now collapse into one chip showing only the canonical base name (`"Blank & Jones"`).

Applied in both the **Playing Now** view and the **Artist Profile** similar artists section. Auto-DJ still receives the full unfiltered list.

---

## Fix: Discogs black placeholder images rejected

Discogs returns a 1203-byte black 400×400 JPEG for artists without a photo. These were being saved and shown as artist images. `saveArtistImage()` now rejects any downloaded image below 5000 bytes silently.

---

## Fix: Unknown artist placeholder image on all surfaces

`unknownartist.webp` is now shown consistently on all four artist image surfaces (Playing Now, Artist Profile, Artist Library letter rows, Artist Home carousel) while a real image is loading or when none is available. All `onerror` handlers fall back to the placeholder rather than showing a broken image.

---

## Fix: Admin hydration panel — activity log count

The "Recent activity" header in the hydration stats panel previously showed `last N` where N was the full array size (up to 20), but the list only ever rendered 10 rows. Now correctly reads "last 10".

---

## Performance: Queue add no longer triggers waveform prefetch

**Before:** Adding 1,000 songs (e.g. an entire album collection) to the queue called `_wfPrefetchEnqueue` 1,000 times, filling a background work queue with 1,000 HTTP requests and server-side ffmpeg jobs. This ran continuously in the background, competing with the waveform canvas animation loop and making the seek bar/cursor sluggish.

**After:** `addSong`, `addAll`, and `playNext` no longer enqueue waveform prefetch at all. Waveforms are fetched:
- When a song **actually starts playing** (`_fetchWaveform` — unchanged)
- For the next 3 upcoming songs when you start a playlist (`_WF_AHEAD=3` — unchanged)
- For the next Auto-DJ song (single prefetch just before playback — unchanged)

Adding 1,000 songs is now a pure in-memory push + one DOM re-render.

## Recently Added — album-art grid + compact view

The Recently Added view now has three density modes (toggled per-user, persisted in localStorage):

- **List** (default) — song rows with album art thumbnails (unchanged)
- **Compact** — song rows without art thumbnails, tighter spacing
- **Grid** — groups songs by album and shows album cards; clicking opens the album, the play overlay starts it immediately
