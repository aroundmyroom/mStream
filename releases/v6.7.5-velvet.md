# v6.7.5-velvet — April 2026 — Crossfade volume-spike fix

## Fix: crossfade volume spikes when outgoing track has a recorded fade-out

### Problem

When a track ends with a few seconds of recorded silence (e.g. 56 s of audio in a 60 s file) and crossfade is set to 12 s, the following three bugs combined to produce a sudden loud "BOEM" at the end of the crossfade:

**1. `_doXfadeHandoff` cancelled the incoming gain curve mid-ramp**
The `ended` event on the outgoing element fires at the true audio end-of-file, which may be several seconds before the `setValueCurveAtTime` sin ramp finishes on the incoming element.  The old code called `cancelScheduledValues(0)` + `gain.value = 1`, which:
- Cancelled all Web Audio scheduled events from time 0 (including the still-running 0→1 sin curve)
- Instantly hard-set the gain to 1.0, causing an abrupt volume spike from wherever the curve was (e.g. 0.6 or 0.7)

**2. `_rgGainNode` was hard-set at handoff time**
`_rgGainNode` is shared downstream of both `_curElGain` and `_nextElGain` in the Web Audio graph.  At handoff, `_applyRGGain(s)` was called immediately, snapping `_rgGainNode` to the new track's ReplayGain value.  If Song B has a higher RG than Song A this is an instantaneous gain jump applied on top of the still-ramping per-element gain — visible as a sudden volume increase exactly when the NP modal, album art, and waveform refreshed.

**3. `cancelScheduledValues(0)` used `currentTime = 0` not `audioCtx.currentTime`**
Calling `cancelScheduledValues` with argument `0` removes ALL events from the beginning of the AudioContext timeline, not just future ones.

### Fix

- **`_startCrossfade`**: added a `linearRampToValueAtTime` on `_rgGainNode` over the full crossfade duration, transitioning from the outgoing track's RG value to the incoming track's RG value in sync with the per-element sin/cos curves.
- **`_doXfadeHandoff`**: removed the `cancelScheduledValues` + `gain.value` hard-set on `_curElGain` (the now-incoming element).  The sin curve scheduled by `_startCrossfade` continues to its natural end on the audio thread — no intervention needed.
- **`_doXfadeHandoff`**: skips `_applyRGGain` when Web Audio is active (the ramp scheduled in `_startCrossfade` already handles the transition); only calls it on the fallback (no Web Audio) path.

The outgoing element (`oldCurGain`) is still silenced instantly at handoff — that element is retired and no longer contributes to output, so an immediate `gain.value = 0` is correct and inaudible.
