# v6.12.3-velvet — Webhooks & Reliability Fixes

**Release date:** April 2026

> Adds generic custom webhook notifications and brings several reliability fixes:
> a Web Worker heartbeat keeps MPV alive in backgrounded tabs, VU meters now stay
> active in cast-to-speaker mode, and Auto-DJ artist exclusion is case-insensitive.

---

## New features

### feat: Custom webhooks — notify any HTTPS endpoint on scrobble
- Admins can now configure up to **2 generic named webhooks** (any `http://` or `https://` URL)
  in **Settings → Webhooks**.
- Webhooks fire a `POST` to the configured URL with a JSON payload on every scrobble:
  ```json
  { "event": "scrobble", "artist": "…", "album": "…", "track": "…", "username": "…", "timestamp": 1234567890 }
  ```
- Users enable/disable each hook individually and can set a display name per slot via the
  player's **Webhooks** panel.
- Discord Webhook is now shown alongside custom webhooks in a single unified **Webhooks**
  panel in both the admin UI and the player, instead of a separate page.

## Bug fixes

### fix: MPV stops playing when browser tab goes to background
- Chrome aggressively suspends `setInterval` in hidden tabs, causing the cast heartbeat to
  stop firing and MPV to time out after 30 seconds.
- Fix: a **Web Worker** sends the heartbeat every 8 s from a separate thread.
  `navigator.locks.request()` (where supported) prevents Chrome from suspending the worker.
  Server-side timeout raised to **5 minutes** — a generous safety net that only fires on
  genuine disconnects (browser kill, network drop), never on normal background tab use.
- A `file-loaded` IPC event now drives seek-on-restore instead of a hardcoded 2.5 s client-
  side delay, so position is applied at the exact right moment regardless of file size.

### fix: VU meter drops to zero after cast-to-speaker + next song
- When casting was active, `audioEl.muted = true` silenced the `MediaElementSourceNode`,
  cutting the signal path to every Web Audio analyser (VU meters, spectrum visualiser).
- Fix: a dedicated `_castMuteGain` GainNode sits **after** the analyser taps in the signal
  chain. Setting `gain.value = 0` mutes the speaker output while keeping analyser data
  flowing. `audioEl.muted` is never set to `true` anymore.

### fix: Auto-DJ artist exclusion was case-sensitive
- SQLite string comparisons are case-sensitive by default, so "Adeva" and "adeva" in the
  cooldown list were treated as different artists, allowing the excluded artist to re-appear.
- Fix: both `ignoreArtists` NOT IN clauses now use `LOWER(f.artist) NOT IN (…)` with the
  exclusion list lowercased client-side before binding.

### fix: Scanner removeStale query skipped rows with NULL sID
- `DELETE FROM files WHERE vpath = ? AND sID != ?` silently kept rows where `sID IS NULL`
  because `NULL != x` evaluates to NULL (not TRUE) in SQL.
- Fix: condition expanded to `(sID IS NULL OR sID != ?)` so NULL-sID stale rows are cleaned.

### fix: On-demand indexed files appeared at top of "Recently Added"
- On-demand index entries used `Date.now()` as their timestamp, pushing freshly-indexed
  files to the top of the Recently Added list even when the file was years old.
- Fix: `ts` is now set from the file's `mtime`, matching the behaviour of the full scanner.

### fix: ffmpeg bootstrap accepted binaries without a known checksum
- When the BtbN release manifest lacked a checksum entry, the bootstrap silently installed
  the binary anyway.
- Fix: a missing checksum is now a hard failure — the download is deleted and an error is
  logged. This prevents a compromised CDN or transient network blip from slipping an
  unverified binary into production.

### fix: AcoustID status endpoint now reports live scan state
- `GET /api/v1/acoustid/status` previously had no way to report whether a scan was running.
- Fix: the response now includes `scanActive: true/false` sourced from `isScanning()` in
  `task-queue.js`.

### fix: Discord webhook config was not persisted via admin UI
- Admin UI read/write endpoints for the Discord webhook were missing; the config was
  effectively read-only after the server started.
- Fix: `GET /api/v1/admin/discord-webhook/config` and `POST /api/v1/admin/discord-webhook/config`
  added to `admin.js`. The POST validates the URL (must be `discord.com` or `discordapp.com`)
  and persists to the config file.
- Discord webhook schema is now validated on boot via the central Joi config schema in
  `src/state/config.js`.
