# v6.6.5-velvet — April 2026

Several small Subsonic refinements: album-list scope respects albumsOnly vpaths,
thumbnail pre-warming on stream start, and on-demand thumbnail generation with
concurrent-request deduplication.

---

## fix: Subsonic album list now respects albumsOnly vpaths

When no `musicFolderId` is provided in `getAlbumList`, the server previously
included every vpath — mixing flat non-album folders (Top-40, Disco, singles, etc.)
into the album list and polluting it with single tracks.

**Fix:** a new `resolveAlbumListScope()` helper now restricts the query to
`albumsOnly` vpaths when no folder is explicitly selected:

- Single albumsOnly root vpath → query that vpath only
- Single albumsOnly child vpath → query into the parent with the correct
  `filepathPrefix` (e.g. `Albums/`)
- Multiple albumsOnly sources → union their DB vpaths
- No albumsOnly configured → fall back to all vpaths (existing behaviour)

Explicit `musicFolderId` always wins and bypasses this logic.

## fix: Subsonic getCoverArt generates thumbnails on-demand with dedup

When a thumbnail (zs- / zl-) was missing and two requests arrived simultaneously
(e.g. track list load + now-playing bar), both could spawn a parallel `sharp`
resize and write to the same file concurrently, occasionally producing a corrupt
image.

**Fix:** a shared `thumbInProgress` map tracks in-flight `sharp` promises.
Concurrent requests for the same thumb await the same promise instead of starting
a duplicate. Thumbnails are generated on-demand at the right size tier and then
served from cache on all subsequent requests.

Thumbnail size tiers revised:

- `size ≤ 160` → `zs-` (92 px, ~4–24 KB) — now-playing bar requests
- `size > 160` → `zl-` (256 px, ~60–170 KB) — full-screen / prefetch requests

(Previous cutoff was `size ≤ 92`; raised to `160` to match iSub's 160-px
now-playing request so it always gets the small fast thumbnail.)

## feat: Subsonic stream pre-warms art thumbnails before audio starts

iOS clients (iSub, Sonora) fire a `getCoverArt` request and a `stream` request
within ~5 ms of each other. On first play the thumbnail did not exist yet, so
`getCoverArt` had to generate it while the audio was already buffering — art
sometimes missed the render window.

The stream handler now pre-generates both `zs-` and `zl-` thumbnails
asynchronously when the song has an `aaFile`, using the same `thumbInProgress`
dedup map. It awaits only the `zs-` tile (the timing-critical one) before
returning audio bytes, ensuring art is available before iOS renders the
now-playing screen.
