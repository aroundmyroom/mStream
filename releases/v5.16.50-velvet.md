# v5.16.50-velvet — April 2026

## Fix: scanner no longer stalls after the first 200 files

### Root cause
`findFilesByPaths()` called `db.exec('BEGIN')` on entry. The `add-file` endpoint
opens its own transaction at the first inserted file and commits it only every
500 inserts. When batch 2 arrived while that transaction was still open, SQLite
raised "cannot start a transaction within a transaction" → HTTP 500 →
"Batch lookup failed" → scanner logged errors for all 200 files in the batch
and stopped permanently.

### Fix
- **`findFilesByPaths()`** — removed `BEGIN/COMMIT` entirely; SELECT reads do
  not need an explicit transaction.
- **`batchUpdateScanIds()`** — replaced `BEGIN/COMMIT` with `SAVEPOINT` /
  `RELEASE` / `ROLLBACK TO`, which are nestable inside any open transaction.

## Feat: "added to DB" counter in the scan progress card

The scan progress card in the admin UI now shows two lines:

- **`N files checked`** — files looked up in the batch endpoint (existing, unchanged)
- **`N added to DB`** *(green)* — files actually inserted into the database,
  updated in real time as insertions happen

Previously the insertion phase was completely invisible: you could see 200
files checked and then nothing for a long time while the DB writes were
happening in the background.
