# v5.16.30-velvet

## What's New

### Albums-Only Folders

#### The Problem

mStream has always been great at discovering music — but it had a blind spot when it came to albums. The system relied entirely on the `album` ID3 tag to decide what counted as an album. That works fine if your library is nothing but studio albums, but real-world music collections are messier.

DJ sets, radio rips, 12-inch singles, compilations, loose tracks filed by label — all of them carry album tags. In a large library, the Albums view could surface thousands of entries that are technically *tagged* as albums but are not albums any reasonable person would want to browse. There was no mechanism to tell the system what actually belongs there.

#### The Solution

You can now mark specific folders as **Albums-Only** in the Admin panel. When one or more folders carry this flag, the Albums browser draws **exclusively** from those folders. Everything else remains fully accessible for playback, Auto-DJ, and search — it simply does not appear in the Albums view.

If no folder has the flag set, the Albums view continues to work exactly as it always has.

**Admin panel → Directories card → `Alb: On / Alb: Off` toggle per folder.** Changes take effect immediately, no rescan needed.

#### For Libraries with Only Albums

If every folder in your library contains albums, flag them all (including child folders of any root). The Albums view will look exactly as before — explicit and consistent.

#### For Mixed Libraries

This is where it shines. Suppose your root vpath is `Music` and underneath it you have:

- `Music/Albums/` — proper studio albums
- `Music/Disco/` — 12-inch edits and single tracks
- `Music/TOP40/` — radio rip collections
- `Music/DJ-Sets/` — recorded sets

Flag `Albums` as Albums-Only. The Albums view now shows only what lives inside `Music/Albums/`. The rest is untouched and fully playable.

Multiple child folders can be flagged — they are all included in the Albums view together.

#### Technical Notes for API Consumers

- `GET /api/v1/ping` now returns `albumsOnly: bool` per vpath in `vpathMeta`.
- `PATCH /api/v1/admin/directory/flags` — set `albumsOnly: true|false`.
- `POST /api/v1/db/albums` and `POST /api/v1/db/artists-albums` — new optional `includeFilepathPrefixes` parameter (whitelist array of `{ vpath, prefix }` objects).

The filtering uses a **hard whitelist** approach (same pattern as Auto-DJ child-vpath filtering): for a parent vpath containing an albums-only child, the SQL becomes `WHERE vpath = 'Music' AND filepath LIKE 'Albums/%'`. Nothing can leak through from other sub-folders or from files sitting directly at the parent root.

## Bug Fix: Albums from non-albumsOnly folders leaking through

The first implementation of the filter used an **exclusion (blacklist)** approach: enumerate all sibling child vpaths that are *not* albums-only and exclude them. This had a fundamental flaw — files stored directly at the parent root (not inside any named sub-folder) and files in sub-folders not registered as vpaths would still appear.

Fixed by switching to a **whitelist** (`includeFilepathPrefixes`) approach throughout:
- New `includePrefixClauses()` helper in `src/db/sqlite-backend.js`
- `getAlbums()` and `getArtistAlbums()` accept a 4th parameter `includeFilepathPrefixes`
- `_albumsOnlyFilter()` in `webapp/app.js` fully rewritten to produce `includeFilepathPrefixes` instead of `excludeFilepathPrefixes`
- `POST /api/v1/db/albums` and `POST /api/v1/db/artists-albums` forward the new parameter from the request body
