# v6.5.2-velvet — April 2026

## fix: localStorage cross-instance contamination

### Problem

When a user's browser had visited one mStream-Velvet instance, all settings (volume, EQ, theme, Auto-DJ config, queue, transcoding prefs) and login credentials (`ms2_token`, `ms2_user`) were stored in localStorage scoped only to the browser origin (protocol + host + port). A second mStream server at the same origin — or a clean reinstall of the same server — would inherit the full localStorage of the previous instance, including pre-filled login credentials that don't belong to the new server.

### Solution

Three-part fix with zero overhead for normal use:

**1. Stable server identity (`src/state/config.js`)**  
On first boot, a `crypto.randomUUID()` is generated and saved to `save/db/default.json` as `instanceId`. This value persists across restarts; it changes only on a clean install (new `default.json`), which is exactly when a wipe is needed.

**2. instanceId exposed on the public ping (`src/server.js`)**  
`GET /api/v1/ping/public` now returns `{ status: 'ok', instanceId: "..." }`. No authentication required — the check can fire before any login form is rendered.

**3. Client-side guard (`webapp/app.js`)**  
An IIFE at the very top of `app.js` fetches `/api/v1/ping/public` immediately on page load:
- **First visit**: stores `instanceId` in `localStorage` as `ms2_server_id`. Nothing else changes.
- **Same server**: `ms2_server_id` matches — proceeds normally.
- **Different / reinstalled server**: mismatch detected → `localStorage.clear()` + `location.reload()`. The user gets a completely clean state as if visiting for the first time — no stale settings, no pre-filled credentials from the old server.

### Behaviour table

| Scenario | Result |
|---|---|
| Same server, same browser | No change, no reload |
| Fresh install at same origin | One invisible reload, then clean state |
| Second server at same origin | Wipe + reload on first visit |
| Server restart (config unchanged) | ID unchanged, no wipe |

### Files changed

- `src/state/config.js` — auto-generate `instanceId` on first boot
- `src/server.js` — expose `instanceId` on `/api/v1/ping/public`
- `webapp/app.js` — server identity guard IIFE at top of file
- `docs/localstorage.md` — `ms2_server_id` documented, Lifecycle + multi-server sections updated
- `docs/API.md` — `/api/v1/ping/public` entry updated
