# v6.7.0-velvet — Album Library enhancements — April 2026

A focused set of quality-of-life improvements to the Album Library: cover art can
now be set from within the album detail view, CUE-sheet albums show all their
individual track titles, and navigating back to the overview is now instant.

---

## feat: album art picker in album detail view (admin only)

Admins can now update the cover art of any album directly from the Album Library
without leaving the page.

- Hover (or tap on touch screens) over the album cover in the detail header to
  reveal a pencil overlay icon; click it to open the art picker.
- **Sources supported:** Discogs search, Deezer search, iTunes search, or paste
  any HTTPS image URL directly.
- The selected image is downloaded, converted to JPEG, and written as `cover.jpg`
  inside the album folder on disk.
- The detail header updates immediately after saving; returning to the album grid
  shows the new art without a page reload (browser cache is automatically
  bypassed).

---

## feat: CUE-sheet albums show individual track titles in the Album Library

Albums that consist of a single FLAC file with an accompanying `.cue` sheet (e.g.
full-CD rips) previously showed only one entry — the FLAC filename — in the track
list. The track list now expands all named tracks from the embedded cue-sheet data
with correct track numbers, titles, and individual durations.

- Clicking a named CUE track loads the FLAC into the queue for that disc and
  seeks directly to that track's start time — playback begins at the correct
  position instantly.
- The existing CUE-marker behaviour in the Now Playing modal and waveform view is
  fully preserved; no changes were made to how CUE points work during playback.

---

## fix: Album Library overview renders instantly on back-navigation

Navigating back to the album grid from a series or album detail view now re-uses
the already-loaded data and renders the grid without a loading spinner or a new
network request. The first load still fetches and shows a spinner as before; the
5-minute server-side cache is respected thereafter.

---

## fix: album cover art path resolution for child vpaths

Albums stored under a dedicated child vpath (e.g. an `Albums/` sub-folder
registered as its own vpath with `albumsOnly: true`) had their cover art path
resolved incorrectly — the sub-folder name was prepended twice, producing a path
like `/media/music/Albums/Albums/Artist/cover.jpg` which never exists.

All three affected code paths have been corrected:

- `resolveArt()` — art discovery at browse time (cover images were not shown in
  the grid at all for these albums)
- `set-art` endpoint — album folder lookup when saving new cover art
- `art-file` endpoint — serving stored cover images by path

---

## fix: replaced cover art now shows immediately in the album grid

After saving new cover art the `art-file` URL includes a cache-bust parameter so
the browser fetches the new image instead of serving the stale cached version.
