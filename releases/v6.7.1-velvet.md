# v6.7.1-velvet — Multi-disc queue overhaul + cover art DB index — April 2026

Two focused improvements: multi-disc albums no longer flood the queue with every
disc at once, and the scanner now stores the cover image filename in the database
so the Album Library can serve art without hitting the filesystem at browse time.

---

## fix: multi-disc albums load only one disc into the queue at a time

Previously, clicking any track on a multi-disc album (e.g. a 4-CD box set) loaded
**all discs** into the queue simultaneously — up to hundreds of tracks for large
compilations. The queue now works disc-by-disc:

### Clicking a track
- If the album is **not yet in the queue**: replaces the queue with **that disc
  only** and starts playback from the clicked track.
- If the album is **already in the queue** (e.g. another disc is currently
  playing): appends that disc to the end of the queue instead of replacing it.

### "Play" and "+ Add to Queue" header buttons
Both buttons now operate on the **currently visible disc tab** only, consistent
with what the user has selected.

### Per-disc "+ " button on each disc tab
Each disc tab is now a grouped pill with a small `+` button on the right:
`[CD 1][+]  [CD 2][+]  …`  
Pressing `+` appends that disc to the end of the queue without switching the
visible tab or interrupting playback.

### Drag-to-reorder whole disc blocks in the queue
The disc separator label (e.g. "CD 1") shown in the queue panel is now draggable.
Dragging it moves **all songs of that disc as a block** to the drop target
position. Individual song drag-and-drop inside a disc is unchanged.

---

## feat: cover image filename indexed in the database

The file scanner now records the discovered cover image filename (`cover.jpg`,
`folder.jpg`, etc.) in a new `cover_file` column on the `files` table. The Album
Library browse endpoint reads this column and returns the `artFile` path directly,
avoiding a filesystem stat on every browse request.

- **Migration**: the column is added automatically on first server start with no
  action required.
- **Populate**: run a full rescan (Admin → Scan) to fill `cover_file` for existing
  library files. Until then the server falls back to the existing art-discovery
  path transparently.
- Named files are preferred in order (`cover.jpg`, `folder.jpg`, `front.jpg`,
  `artwork.jpg`) before falling back to the first image found; this matches the
  existing scanner behaviour.
