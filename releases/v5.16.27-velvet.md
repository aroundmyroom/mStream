## v5.16.27-velvet — 2026-03-24

### Radio recording & scheduling, ListenBrainz, Docker, podcast save-to-library, Audio Content view, recording modal, Auto-DJ pills

This is a large cumulative release covering six major features, multiple quality-of-life improvements, and a range of fixes since v5.16.26-velvet.

---

### Radio Stream Recording

Users with the `allow-radio-recording` permission (granted by admin per-user) can record any live radio stream directly to disk.

- Red `●` record button appears in the playbar while a radio stream is playing and the user has the permission — pulses during active recording with an elapsed-time pill
- Clicking the button opens a modal listing all configured `recordings`-type vpaths; user selects a destination and clicks **Start Recording**
- Recording stops automatically when the user switches to a non-radio track
- File naming: `{StationName}_{YYYY-MM-DD_HH-MM-SS}.{ext}` — extension resolved from the `Content-Type` header
- Write-permission probe before starting (temp file create/delete)
- SSRF protection on all stream URLs
- **Cover art embedded** — after recording closes, FFmpeg embeds the station logo as attached picture (mp3, m4a, aac, flac); best-effort, recording is never lost on FFmpeg failure
- **Max recording duration** — admin-configurable cap (`scanOptions.maxRecordingMinutes`, default 180 min); set in the DB Scan Settings card → "Max Recording Duration" modal
- **New vpath type** `recordings` — folders marked as Recordings are excluded from all library scans
- **Admin toggle** — `POST /api/v1/admin/users/allow-radio-recording` enables the feature per user
- **New endpoints:** `GET/POST /api/v1/radio/record/active`, `POST /api/v1/radio/record/start`, `POST /api/v1/radio/record/stop`, `POST /api/v1/admin/users/allow-radio-recording`, `POST /api/v1/admin/db/params/max-recording-minutes`

### Scheduled Radio Recording

Users can schedule radio streams to record automatically at a specific time, with recurrence.

- **Record modal redesigned** — two tabs: **Record Now** (existing immediate flow) and **Schedule** (form + schedule list)
- **Recurrence modes**: Once, Every day, Weekdays (Mon–Fri), Custom (choose specific days)
- **Server-side ticker** (`src/api/radio-scheduler.js`) — 30 s interval fires scheduled recordings even when the browser is closed; `once` schedules auto-disable after firing
- **DB table** `radio_schedules` — per-user; stores station, URL, art, vpath, start time/date, duration, recurrence, enabled flag
- Schedule list shows On/Off toggle and delete button; active recordings show a pulsing "recording" indicator
- **Live system clock** shown in the modal — "System date & time: Mon 24 Mar 2026  16:42:07" so users see exactly what the server considers "now"
- **New endpoints:** `GET/POST /api/v1/radio/schedules`, `DELETE /api/v1/radio/schedules/:id`, `PATCH /api/v1/radio/schedules/:id/enable`

### ListenBrainz Scrobbling + Now-Playing

Full ListenBrainz integration alongside existing Last.fm support.

- Admin enables ListenBrainz server-wide; each user enters their own token in a new **ListenBrainz** nav page
- Token validated against LB `/1/validate-token` on connect
- Scrobble fires 30 s after track start (`listen_type: "single"`) — same trigger as Last.fm
- **Instant "Listening Now" ping** — `POST /api/v1/listenbrainz/playing-now` fires immediately on track start; current track appears on the LB dashboard in real time
- Both LFM and LB can run simultaneously; scrobble status badge shows combined "Last.fm ✓ · ListenBrainz ✓"
- **New endpoints:** `GET/POST /api/v1/admin/listenbrainz/config`, `GET /api/v1/listenbrainz/status`, `POST /api/v1/listenbrainz/connect`, `POST /api/v1/listenbrainz/disconnect`, `POST /api/v1/listenbrainz/scrobble-by-filepath`, `POST /api/v1/listenbrainz/playing-now`

### Podcast: Save Episode to Library

Each episode row now has a **Save to Library** button (↓ icon) that downloads the episode audio from the CDN to the server.

- Saved to `<AudioBooks vpath>/<feed title>/<YYYY-MM-DD Episode Title>.ext` — subfolder created automatically
- Streamed directly to disk using `node:stream pipeline` — no RAM buffering, safe for large files
- Button states: idle → spinning → green ✓ (success toast) or red ✕ (error toast); resets to idle after 4 s
- SSRF protection, path sanitisation, partial-file cleanup on error
- **New endpoint:** `POST /api/v1/podcast/episode/save`

### Subsonic API Stubs

Added empty-but-valid stub responses for `getArtistInfo`, `getArtistInfo2`, `getAlbumInfo`, `getAlbumInfo2`, `getSimilarSongs`, `getSimilarSongs2`, `getTopSongs`. These had no router entry at all — Subsonic clients (Symfonium, DSub) would receive unhandled-method errors and retry aggressively. Stubs return well-formed empty responses so clients accept the result and stop retrying.
