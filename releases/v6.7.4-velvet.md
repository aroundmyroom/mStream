# v6.7.4-velvet — April 2026 — Mute fix, album source filter, art thumbnails

## Fix: mute icon and server state are correctly cleared when dragging the volume slider up

Three separate bugs combined to break the mute experience:

**1. Dead second `input` listener**
The slider had two `input` listeners registered. The second one was supposed to clear the mute icon when the user dragged the slider away from 0, but its guard condition (`audioEl.volume === 0`) was always false by then — because the first listener had already set `audioEl.volume = value / 100`. So the mute icon stayed red and `localStorage('mute')` was never cleared.

**2. Server retained `mute='1'` after slider drag**
Because the localStorage key was never cleared (bug 1), the 300 ms `_syncPrefs()` call from the first listener still sent `mute: '1'` to the server. The server then permanently held the "muted" state.

**3. Tab switch re-muted audio**
Switching away and back to the tab triggers a `visibilitychange` → server settings fetch → `_applyServerSettings()`. Since the server still had `mute='1'` (bug 2), it forced `audioEl.volume = 0` on every tab switch, stopping playback even though the user had visually unmuted.

**Fix:** Merged the dead second `input` listener into the first. Mute clearing now checks `mute-btn.classList.contains('muted')` (reliable). Added a guard in `_applyServerSettings`: server `mute='1'` is ignored when local state says unmuted, preventing tab-switch re-mutes while the sync push is in-flight.

## Feat: Album Library source filter pills

When two or more albumsOnly folders are configured (e.g. **Albums** and **Disco**), the Album Library header now shows toggle pills — one per source folder. Clicking a pill shows or hides albums from that folder. Both are on by default. The text search only matches albums within the currently enabled sources.

- Server: `sourceVpath` field added to every album and series object in `/api/v1/albums/browse`
- Client: pills rendered using the existing `.dj-vpath-pill` style from the Search tab; hidden when only one source exists

## Fix: on-demand art and Now Playing art no longer served at full resolution

Two image-serving gaps fixed:

**Gap 1 — `GET /api/v1/files/art` never generated thumbnails**
When art was extracted on-demand (player bar, queue, file explorer), the raw original was written to the cache but `zl-` (256 px) and `zs-` (92 px) thumbnails were never created. The player then immediately requested `?compress=l`, fell through to the full-res original, and served a multi-MB image where a 12 KB thumbnail was expected. Fixed: thumbnails are now generated immediately after the original is written.

**Gap 2 — `compress=m` always returned full-res**
`zm-` (512 px, used for the Now Playing modal) was listed in the skip-list of the force-recompress script but was never actually generated anywhere — not by the scanner, not on-demand, not by force-recompress. Fixed: `zm-` is now generated in all three places.

Existing originals in the cache will receive `zm-` thumbnails via **Admin → Force Recompress Images**.
