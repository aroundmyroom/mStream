## Search performance fix

**Symptom**: Users with large libraries (300 k+ files) or on spinning-disk servers reported search taking 15+ seconds, sometimes hanging indefinitely.

**Root cause** â€” three compounding problems in the FTS5 query used by `searchFiles` and `searchFilesAllWords`:

1. **`ORDER BY bm25(fts_files)`** â€” using the auxiliary `bm25()` function instead of the special `rank` column disabled FTS5's top-N early-stop optimisation. SQLite had to materialise and rank *all* matching rows before applying LIMIT. For a common word like "love" in a 300 k-file library that can be tens of thousands of rows.

2. **Wrong JOIN order** â€” `FROM files f JOIN fts_files ft` put the `files` table as the outer loop. The SQLite FTS5 documentation explicitly states that the FTS table must be the outer (driving) table for the index scan to be used efficiently.

3. **`LIMIT 500`** â€” the client already slices song results to 50 before rendering them. Fetching 500 from the server was 10Ă— wasted work on both server and network.

**Fix**:

```sql
-- Before (slow)
SELECT f.rowid AS id, f.* FROM files f
  JOIN fts_files ft ON f.rowid = ft.rowid
  WHERE f.vpath IN (?) AND ft.fts_files MATCH ?
  ORDER BY bm25(fts_files) LIMIT 500

-- After (fast)
SELECT f.rowid AS id, f.* FROM fts_files ft
  JOIN files f ON f.rowid = ft.rowid
  WHERE f.vpath IN (?) AND ft.fts_files MATCH ?
  ORDER BY rank LIMIT 50
```

**Measured results** on our 134 k-file library (live server, first-request cold):

| Query | Time |
|-------|------|
| `"love"` | 157 ms |
| `"happy"` | 32 ms |
| `"the"` | 201 ms |
| `"amsterdam"` | 10 ms |

The same queries with the old code hung indefinitely (>30 s) even locally.

## Auto-DJ artist cooldown â€” dot-stripping

**Bug**: "M.C. Sar" and "MC Sar" were treated as different artists by the Auto-DJ cooldown logic, so the same artist could appear back-to-back if their name was stored inconsistently across tracks.

**Fix**: Both the SQL `ignoreArtists` clause and the client-side `_djPushArtistHistory` deduplication now strip dots before comparison:

- SQL: `REPLACE(LOWER(f.artist), '.', '') NOT IN (...)`
- Client: `const norm = a => a.trim().toLowerCase().replace(/\./g, '')`
