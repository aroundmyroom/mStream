# v6.12.10-velvet — April 2026 — Search Completeness & Accuracy

## Overview

This release fixes a cluster of bugs that caused search results to be incomplete or wrong for users with large, well-organised libraries. The most visible symptom was searching for an artist like "Cerrone" and only seeing 1–4 albums instead of the full 170+. All result categories (Artists, Folders, Albums, Tracks) now return accurate, unfiltered counts.

---

## Fix: Album search — finds all albums BY the artist, not just albums named after the query

**Symptom**: Searching "Cerrone" in Albums returned ~4 results instead of 170+.

**Root cause**: The old code called `searchFiles('artist', ...)` and deduped in JavaScript. That function has `LIMIT 50` on the SQL rows. For Cerrone (1 410 tracks), the top 50 FTS-ranked rows all came from 2–3 heavy-track albums (Dream = 57 tracks, Hysteria = 37 tracks) so JS deduplication only ever saw those albums.

**Fix**: New `searchAlbumsByArtist()` function that runs `GROUP BY f.album` at the SQL level, so the limit (now removed entirely) would have counted unique albums rather than individual tracks. Uses `MAX(aaFile)` / `MAX(cover_file)` to ensure art is never lost.

---

## Fix: All search categories now return complete unfiltered counts

**Symptom**: Tab bar showed e.g. "Albums (49)" when there were 174 actual Cerrone albums; "Folders (50)" when there were 105 matching folders.

**Root cause**: Every search backend function had a hard `LIMIT 50` (folders, tracks, title, files) or `LIMIT 200` + `.slice(0, 50)` (artists). These caps silently truncated results before they ever reached the client.

**Fix**: Removed all `LIMIT` caps from `searchFiles`, `searchFilesAllWords`, `searchFolders`, `searchArtists` prepared statement, and `searchArtistsNormalized`. The frontend already renders max 50 track rows to DOM (with an overflow note showing the real total) — so display performance is unchanged.

**Result for "cerrone"**: artists: 10 | folders: 105 | albums: 174 | tracks: 1 470

---

## Fix: Multi-disc albums — clicking from artist profile showed only first disc

**Symptom**: Albums with `CD1/` and `CD2/` sub-folders only loaded the first disc's tracks when opened from the artist profile page.

**Root cause**: `viewArtistProfile`'s album click handler passed `alb.dir` (the raw SQL dir like `Pink Floyd at Pompeii/CD1/`) to `viewAlbumSongs`. The DB query `filepath LIKE 'Pink Floyd at Pompeii/CD1/%'` matched only CD1 tracks.

**Fix**: Use `alb.normDir || alb.dir` — `normDir` is the `_normaliseAlbumDir`-collapsed parent path (`Pink Floyd at Pompeii`) that covers all discs.

---

## Fix: Album art — cover.jpg in album root not found for multi-disc tracks

**Symptom**: Tracks inside `Album/CD1/` showed no album art even when `Album/cover.jpg` existed.

**Root cause**: `checkDirectoryForAlbumArt()` only scanned the track's immediate directory. If `CD1/` had no image files it returned `false` without checking the parent.

**Fix**: Added one-level parent directory fallback: if the immediate directory has no images, scan `path.dirname(directory)` (bounded by the vpath root) and push any found images as `../filename` relative paths.

---

## Fix: Artist profile album list — album art missing despite art existing

**Root cause**: `getArtistAlbumsMulti` and `getArtistAlbums` used bare `aaFile` and `year` as SELECT columns inside a `GROUP BY` query. SQLite picks an indeterminate row from each group — which could have `aaFile = NULL` even when other rows in the group had art.

**Fix**: Changed to `MAX(aaFile) AS album_art_file`, `MAX(cover_file) AS cover_file`, `MAX(year) AS year` so non-null values always win.

---

## Fix: SQLite performance — statement cache, mmap, ANALYZE, page_size migration

Four independent improvements that reduce query latency on large libraries (134k+ files):

- **Statement cache** (`_stmtCache` Map + `_prepare(sql)` helper): dynamic search SQL is compiled once and reused across keystrokes instead of re-compiled on every search.
- **`PRAGMA mmap_size = 134217728`** (128 MB): SQLite maps the database file into address space; OS page cache is shared directly instead of a memcpy on every page read.
- **Deferred `ANALYZE` on startup**: populates `sqlite_stat1` with accurate row counts so the query planner makes optimal index decisions for FTS JOIN queries. Docker deployments with multiple music roots were flying blind without this.
- **Fixed page_size migration**: the upgrade from 4 KB to 8 KB pages was silently skipped because SQLite cannot change `page_size` in WAL mode. Fix switches to `journal_mode=DELETE` before VACUUM then restores WAL. Page size is now confirmed 8192.
