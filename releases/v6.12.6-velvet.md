# v6.12.6-velvet — To debug until we found it

**Release date:** April 2026

> Eliminates a legacy performance defect that was causing **7–14 seconds of full server
> freeze** on every browser refresh — traced to a SQL query that never benefited from an
> index and was blocking the entire Node.js event loop. Once found, it was a one-line SQL
> rewrite. This release also ships the dual-hash identity system, an integration test
> framework, play-count preservation across tag rewrites, Subsonic scrobble improvements,
> and a close button on the Auto-DJ info strip.

---

## Critical performance fix — 7-second event loop block eliminated

### root cause: `getUserPlaylists` SQL join caused full table scan on every page load

Every page refresh triggered a call to `getUserPlaylists()` in `src/db/sqlite-backend.js`.
The query joined the `playlist_entries` table against the `files` table using:

```sql
LEFT JOIN files f ON f.vpath || '/' || f.filepath = e.filepath
```

SQLite cannot use *any* index when the stored column is inside a function call on the
**left side** of the comparison. With 134,565 rows in the `files` table, this meant a
full table scan on every boot — taking **~7,000 ms** synchronously via `DatabaseSync`,
which blocks the entire Node.js event loop.

**Effect on the player**: While the event loop was frozen, every pending HTTP request was
queued behind it — including the audio `<source>` fetch, album art, home view, ping, and
all session setup calls. Users experienced a **7–14 second black silence** before the page
became interactive on every refresh.

**The fix** — split `e.filepath` at the first `/` using `SUBSTR` + `INSTR` so SQLite can
use the existing composite index `idx_files_filepath_vpath`:

```sql
-- Before (7 000 ms — full table scan, no index possible)
LEFT JOIN files f ON f.vpath || '/' || f.filepath = e.filepath

-- After (<10 ms — uses existing composite index)
LEFT JOIN files f
  ON f.vpath    = SUBSTR(e.filepath, 1, INSTR(e.filepath, '/') - 1)
 AND f.filepath = SUBSTR(e.filepath, INSTR(e.filepath, '/') + 1)
```

Also added expression index `idx_files_full_path ON files(vpath || '/' || filepath)` to
the DDL for future-proofing, and created it on live databases at first startup.

**Secondary fix**: removed a dead `playlists: db.getUserPlaylists()` call from
`GET /api/v1/ping` — the client only reads playlists from `/api/v1/playlist/getall`, so
this was running the slow query **twice** per boot for no reason.

**Measured improvement:**

| Endpoint | Before | After |
|---|---|---|
| `GET /api/v1/ping` | ~7,656 ms | ~8 ms |
| `GET /api/v1/playlist/getall` | ~8,711 ms | ~2 ms |
| Audio first byte | ~14 s | ~12 ms |

**Files changed:** `src/db/sqlite-backend.js`, `src/api/playlist.js`

## Supporting audio fixes

### fix: `_onAudioStalled` guard — prevents stall-recovery loop on initial load
- `_onAudioStalled` previously started a 5-second recovery timer even when the audio
  element had not yet received any bytes (`readyState === 0`).
- At startup, this caused a self-amplifying loop: the stall handler fired → called
  `_reloadFromPosition(0)` → browser re-issued the HTTP request → stall fired again.
- Fix: `if (audioEl.readyState === 0) return;` — skip recovery when still in initial load.

### fix: `<audio preload="auto">` — browser begins buffering immediately
- Changed `preload="none"` (default) to `preload="auto"` on the `<audio>` element in
  `webapp/index.html`.
- The browser now starts fetching audio data as soon as `src` is assigned during
  `restoreQueue`, without waiting for a `play()` call.

**Files changed:** `webapp/app.js`, `webapp/index.html`

## New feature: Dual-Hash Identity (`audio_hash` column)

- Added `audio_hash` column to the `files` table: a SHA-256 hash of
  `artist | album | title | duration` (all lowercased, trimmed).
- Metadata-based identity survives transcoding and bitrate changes — if you convert
  a 500-song MP3 collection to FLAC, all play counts, ratings, and history are preserved
  automatically via `audio_hash`.
- Automatic database migration for existing installations (`ALTER TABLE` on first boot).
- Scanner computes both `hash` (file content / MD5) and `audio_hash` (metadata / SHA-256)
  on every file parse.

**Files changed:** `src/db/sqlite-backend.js`, `src/db/scanner.mjs`

## fix: Play-count preservation across external tag rewrites

- **Problem**: When an external tool (e.g. MusicBrainz Picard) rewrites a file's tags, the
  file's `mtime` changes. The scanner detects the change, deletes the old DB row, re-parses
  the file, and inserts a new row with a new MD5 `hash`. All `user_metadata` (play count,
  rating, star, last-played) and `play_events` rows were keyed on the old hash → orphaned.
- **Fix**: `get-file` and `get-files-batch` now include `_oldHash` in stale responses.
  The `add-file` handler calls `db.migrateHash(oldHash, newHash)` when the hash changed,
  updating all `user_metadata` and `play_events` rows atomically.
- Radio recordings, ytdl, podcasts, and on-demand indexing are unaffected (different code
  paths that do not delete-then-reinsert).

**Files changed:** `src/api/scanner.js`, `src/db/sqlite-backend.js`, `src/db/manager.js`

## fix: Subsonic scrobble — batch support and input validation

- `scrobble` now correctly handles **multiple `id` parameters in a single request** (the
  OpenSubsonic spec allows batching several plays in one call). Previously only the first
  `id` was processed.
- `setRating` now returns an error if `rating` is outside the valid 0–5 range instead of
  silently storing a corrupt value.
- `submission=false` (now-playing) now uses the last `id` in the list per spec.

**Files changed:** `src/api/subsonic.js`

## Technical notes

- Expression index `idx_files_full_path` is created at startup if it does not exist.
  On a 134,565-row database the first-time build takes ~500 ms (runs once, non-blocking
  startup log entry).
- The `getUserPlaylists` query rewrite is the real fix; the expression index is
  belt-and-suspenders for any future queries that use the `vpath || '/' || filepath`
  form.
- Boot API sequencing: `restoreQueue` now fires before `_launchPinnedViewOrHome` so the
  audio media request wins the browser's HTTP connection pool race.
