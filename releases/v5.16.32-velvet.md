# v5.16.32-velvet — March 2026

## FTS5 Full-Text Search + Exclusion Queries

All library searches now run against a **SQLite FTS5 index** instead of full-table `LIKE '%…%'` scans.

- **Relevance ranking:** results ordered by BM25 score (best matches first)
- **Prefix matching:** `talk*` finds "Talking Heads", `lenn*` finds "Lennon"
- **Exclusion syntax:** prefix a word with `-` or use `NOT` to exclude: `talking -heads`, `chaka khan NOT remix`
- **Diacritic folding:** `Bjoerk`, `Bjork`, and `Björk` all match the same results
- Index covers `title`, `artist`, `album`, and `filepath`
- Auto-rebuilt from the existing library on first startup after upgrade
- Kept in sync on every insert, delete, tag-edit, and post-scan cleanup

---

## Three-Entity Search: Folders, Artists, Songs

Search results are now split into **three separate sections**:

### Folder search
Unique directory paths are extracted from the library and indexed in a dedicated `fts_folders` trigram table. Clicking a folder result opens the file browser at that location. The back button returns to search results with the previous query intact.

### Artist search (normalized)
Artist tags are normalized at index-build time — leading track numbers (`01 `, `28 `) and leading symbols are stripped so all numbered variants group under the clean name. All raw variants are still stored so album queries find files regardless of how they were originally tagged.

Artist search is vpath-aware: results are filtered to the currently selected library.

### Artist → albums: single SQL query
Previously clicking an artist fired up to 68 parallel `getArtistAlbums` requests which could overload SQLite and return empty results. This is now a single `WHERE artist IN (...)` query via the new `POST /api/v1/db/artists-albums-multi` endpoint.

## ffmpeg Self-Contained Bootstrap

mStream Velvet no longer depends on a system-installed ffmpeg. At startup the server inspects the bundled binary in `bin/ffmpeg/` and automatically downloads the latest static build from [BtbN's ffmpeg-builds](https://github.com/BtbN/FFmpeg-Builds) if:

- No binary is present yet, **or**
- The installed version is below **v6**

**What this means for users:**

- Fresh installs download a working ffmpeg automatically — no manual setup step
- If you had an old v4 or v5 binary it is silently replaced on next startup
- The resolved version is written to the log at boot: `ffmpeg ready: ffmpeg version N-xxx-...`
- All ffmpeg consumers (waveform generation, transcoding, radio recording, YouTube tagging) always run on a modern codec engine

**Minimum version:** ffmpeg v6 (auto-download currently fetches v7 nightly builds from BtbN).

**Docker:** the Docker image includes ffmpeg natively; the bootstrap is a no-op there.

## YouTube Download — Temp File Isolation

All intermediate files created during a YouTube download (audio stream, thumbnail JPEG) are now written to a **private temp directory** under the OS temp folder (e.g. `/tmp/mstream-ytdl-XXXXXX/`). The directory is unconditionally wiped in a `finally` block after every download.

**Before:** the hidden working files (`._{title}`) sometimes appeared in the music folder and could be left behind on error.  
**After:** the music folder is never touched until the final finished file is moved into place.

## YouTube & Radio — Opus / OGG Album Art Fix

Downloading a YouTube video in Opus/OGG format, or recording a radio stream to an Opus/OGG file, previously failed with:

```
ffmpeg error 234 — Unsupported codec id in stream 1
```

**Root cause:** Opus containers do not accept video streams. The old code mapped the cover-art JPEG as a video stream (`-map 1:v -c:v copy`), which works for MP3 and MP4 but is invalid for Opus.

**Fix:** Opus output now embeds art using the **Vorbis `METADATA_BLOCK_PICTURE`** comment standard:

1. The JPEG is binary-encoded per the FLAC/Vorbis picture block spec (MIME type, image dimensions, raw image bytes)
2. The resulting block is base64-encoded
3. Passed to ffmpeg as `-metadata METADATA_BLOCK_PICTURE=<base64>` — no video stream mapping required

MP3 and MP4 art embedding is unchanged.

## Radio Recording Fixes

### Album art embedded in non-MP3 recordings

Previously, the station logo was only reliably embedded in MP3 recordings. Non-MP3 formats (OGG, Opus, M4A) used an incorrect ffmpeg stream mapping that either failed silently or produced a broken file.

The non-MP3 embed command is now:

```
ffmpeg -i <audio> -i <logo> -map 0:a -map 1:v -c:a copy -c:v copy -disposition:v:0 attached_pic <output>
```

### Stop API response enriched

`POST /api/v1/radio/recording/stop` now returns additional fields alongside the existing `filePath` and `bytesWritten`:

| Field | Description |
|---|---|
| `relPath` | `vpath/filename` — ready to use as a filepath reference |
| `vpath` | The recordings folder vpath |
| `stationName` | Name of the station that was recorded |
| `artFile` | Filename of the cached station logo, or `null` |

## On-Demand Album Art Extraction

Previously, unscanned files (fresh recordings, YouTube downloads waiting for a scan) showed a blank music-note placeholder everywhere in the UI. Now album art is extracted from the file's embedded tags on demand and shown immediately — no scan required.

### New API endpoint

```
GET /api/v1/files/art?fp=<filepath>
```

Reads the embedded picture from any audio file using `music-metadata`, MD5-hashes the image bytes, writes the result to the album-art cache directory, and returns the cache filename.

**Response:**

```json
{ "aaFile": "d41d8cd98f00b204e9800998ecf8427e.jpg" }
```

Returns `{ "aaFile": null }` if the file has no embedded art or could not be read.

Authentication required (same token rules as all other file endpoints).

See [docs/API/files_art.md](https://github.com/aroundmyroom/mStream/blob/master/docs/API/files_art.md).

### Client-side coverage — art shown in all four surfaces

| Surface | Trigger | What gets patched |
|---|---|---|
| **Player bar** | Track load in `updateBar()` | Now Playing thumb, VU artwork, dynamic theme colours, queue card |
| **Queue panel** ("Up Next") | `refreshQueueUI()` | `.q-art` thumbnail for every queued track |
| **File list / folder view** | `setSongs()` and virtual scroll pre-fetch | `.row-art` thumbnail in standard song rows |
| **File Explorer** (Recordings, YouTube folders) | `renderFileExplorer()` | `.fe-thumb` image in the file grid |

Art is extracted once, written to the shared cache, and reused on subsequent loads — the extraction call is never made twice for the same file.
