# v5.16.47-velvet — April 2026

## fix: scanner batch — correct transaction API for node:sqlite

Rescans were still producing "Batch lookup failed: Request failed with status code 500" errors after v5.16.46.

**Root cause:** `findFilesByPaths()` and `batchUpdateScanIds()` in `sqlite-backend.js` were calling `db.transaction()` — a `better-sqlite3` API that does not exist on Node.js's built-in `node:sqlite` `DatabaseSync`. This threw `TypeError: db.transaction is not a function` on every batch call, causing every rescan batch to fail and accumulate errors in `scan_errors`.

**Fix:** Both functions now use `db.exec('BEGIN')` / `db.exec('COMMIT')` / `db.exec('ROLLBACK')` — the correct transaction pattern for `node:sqlite` `DatabaseSync` already used throughout the rest of the file.

**Cleanup:** The 82,000 stale "Batch lookup failed" entries accumulated in `scan_errors` were deleted.

## fix: initial scan no longer stalls 10+ minutes before processing files

On a first-ever scan (empty database), the scanner was running a full separate directory walk (`countValidFiles`) before doing any actual scanning — purely to get a file count for the progress bar. For a 138K-file library this doubled all I/O and added 10+ minutes of dead time.

**Fix:** Eliminated the pre-count pass entirely. Scanning now starts immediately. A `_totalSeen` counter increments as `recursiveScan` discovers files, and a single `set-expected` ping after the walk sends the accurate total to the progress bar. No double-traverse.
