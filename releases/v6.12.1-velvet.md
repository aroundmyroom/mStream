# v6.12.1-velvet — DLNA Polish & Background Fix

**Release date:** April 20, 2026

> Fixes and polish on top of the v6.12.0 DLNA launch: empty browse containers, unreliable
> auto-start on restart, and the AcoustID fingerprinter starving the audio stream of disk I/O.

---

## Bug fixes

### fix: DLNA browse — albumsOnly containers empty after server restart

After a server restart, browsing into any `albumsOnly` vpath (e.g. Disco, Albums) returned 0
items even though the albums were clearly in the database.

**Root cause:** `resolveAlbumsSources()` returns a filepath prefix that already includes a
trailing slash (e.g. `"Disco/"`). The browse handler was appending another `/` when building
the SQL `LIKE` filter → `"Disco//"` → no rows matched.

**Fix:** Use `source.prefix` directly without appending an extra separator. Affects both the
`src_` folder-node handler and the `dir_` source-finder that locates which source a
sub-container belongs to.

### fix: DLNA auto-start unreliable on rapid service restart

When `systemctl restart music.service` was run, port 10293 could still be in `TIME_WAIT` state
from the previous process. The DLNA `start()` call would throw `EADDRINUSE`, silently log a
warning, and leave DLNA stopped — even though `enabled: true` was correctly in the config file.
The admin UI would show the orange "Enabled (not running)" badge with no indication of when it
would recover.

**Fix:** The boot wiring now retries `start()` automatically after 5 seconds if the first
attempt fails. The second attempt always succeeds once the OS releases the port.

### fix: AcoustID fingerprinting caused audio stream stalls

With AcoustID enabled (`autostart: true`) every server restart re-triggered the background
fingerprinting worker. `fpcalc` reads up to 120 seconds of audio from disk per file — the same
disk bus used to serve audio streams to browsers. Running at default I/O priority, `fpcalc`
starved the stream of disk bandwidth, causing the browser's audio buffer to run dry and playback
to pause silently (no console error, just `readyState=0`).

**Fix:** `fpcalc` is now launched via `ionice -c 3 nice -n 19` — idle I/O scheduling class and
lowest CPU priority. The OS will preempt `fpcalc` whenever any other process (including the audio
stream) needs the disk. Confirmed: `ionice -p <pid>` reports `idle` and `ps` shows `N` flag.

## Improvements

### feat: DLNA notifies devices after every library rescan

After a completed library scan, `bumpSystemUpdateID()` is now called automatically. DLNA clients
that have subscribed via GENA will receive a `ContainerUpdateIDs` event and refresh their own
catalogue — no manual "rescan" required on the TV or player side.

### feat: DLNA auto-start retry on port busy

As described above — DLNA now retries once after 5 s on EADDRINUSE instead of staying stopped.

### perf: Home view — progressive shelf loading

The Home view previously issued all four API calls in a single `Promise.all` and only rendered
once all four had completed (typically 250–350 ms until anything appeared). Each shelf now
renders independently as its API call resolves:

| Shelf | Typical latency |
|-------|----------------|
| Greeting (no API required) | instant |
| Mood shelves (most-played) | ~44 ms |
| Continue Listening (recently-played) | ~275 ms |
| Recently Added albums | ~256 ms |
| On This Day / stats strip | ~258 ms |

On slower connections (proxy, mobile) the page feels instant rather than blank for several
hundred milliseconds.

### fix: DLNA docs — VLAN / multi-subnet guide added

Added a dedicated **VLANs and network segmentation** section to [docs/dlna.md](../docs/dlna.md)
explaining why DLNA discovery fails across VLANs (SSDP multicast is Layer-2 scoped), with a
solutions table covering: same VLAN, inter-VLAN routing, IGMP proxy, and manual server entry.
UniFi / pfSense / OPNsense specific guidance included. Troubleshooting table updated with
VLAN-aware rows.
