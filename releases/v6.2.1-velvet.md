# v6.2.1-velvet — April 2026

Bugfix release for the Your Stats (Wrapped) feature introduced in v6.2.0.

## Bug fixes

### fix: Your Stats — play-start not fired after server restart / page reload
Auto-resumed playback (queue restored from localStorage) and manual resume via the play button both called `audioEl.play()` directly without going through `playAt()`, so `play-start` was never sent to the server. All listening after a server restart or page reload was invisible to the stats tracker. Both paths now fire `play-start` correctly.

### fix: Your Stats — playedMs overcounted on resumed tracks
When a song resumed from a saved position (e.g. at 1:05), `playedMs` was calculated from `audioEl.currentTime` / `audioEl.duration` without subtracting the resume offset. A 2-minute listen on a 7-minute track would be logged as 7 minutes. A new `_wrappedTrackStartOffset` variable tracks the position at which tracking began; all `play-end`, `play-skip`, `play-stop`, and `beforeunload` beacon payloads now subtract it.

### fix: Your Stats — duplicate play counts when song hash exists in multiple folders
The `getWrappedDataInRange` SQL query joined `play_events` against `files` without deduplication. Because many tracks exist in multiple folders (compilations, mixes) with identical audio hashes, each play event returned multiple rows — inflating play counts and listening time. Fixed by grouping the `files` sub-join on `hash` so each event always maps to exactly one metadata row.

### fix: folder cover art (cover.jpg / folder.jpg) not shown in file explorer
The `/api/v1/files/art` endpoint only extracted embedded cover art from audio file tags. Songs without embedded art but with a `cover.jpg` or `folder.jpg` in the same directory showed no art. The endpoint now falls back to scanning the song's directory for common image filenames (`cover.jpg`, `cover.png`, `folder.jpg`, `folder.png`, `front.jpg`, `front.png`, `artwork.jpg`, `artwork.png`) and caches the first match to the album-art directory.
