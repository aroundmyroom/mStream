# v6.11.1-velvet — Queue clear bug fix

**Release date:** April 18, 2026

## Bug fixes

### Queue clear not persisted across page reload

Pressing the **clear queue** (trash) button correctly emptied the visible queue, but refreshing the page brought all songs back.

**Root cause — three separate issues:**

1. **`_applyServerSettings()`** — ran at every page boot and wrote the DB queue unconditionally to localStorage, including stale non-empty data fetched before the cleared queue had been flushed to the DB (2-second debounce window).

2. **`restoreQueue()`** — bailed out silently (`return`) when the stored queue array was empty, so it never updated `S.queue = []` or called `refreshQueueUI()`.  
   Empty-array entries in localStorage were therefore harmless no-ops, leaving the old in-memory queue untouched after a DB-overwrite.

3. **Clear button debounce** — the DB flush was delayed up to 2 seconds.  An F5 within that window would get the old queue from the DB.

**Fixes:**

- Clear button now flushes to the DB **immediately** (no debounce), so the DB reflects the empty queue before any reload.
- `restoreQueue()` now explicitly sets `S.queue = []; S.idx = -1; refreshQueueUI()` when the stored queue is empty.
- `_applyServerSettings()` only writes a non-empty DB queue to localStorage (empty-queue state is already correctly owned by `persistQueue()` which writes immediately to localStorage on clear).
